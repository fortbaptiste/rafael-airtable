Jour 1

La structure

Le premier jour, je pose les fondations. C'est la partie qui n'est pas visible pour le client mais qui fait que tout tient debout derriere.

Je commence par creer les tables. En general, un projet client a entre 3 et 8 tables selon la complexite. Un CRM classique, c'est une table Contacts, une table Entreprises, une table Deals, une table Activites, une table Produits ou Services. Un calendrier editorial, c'est une table Contenus, une table Plateformes, une table Campagnes. Un suivi de prospection, c'est une table Prospects, une table Relances, une table Setters, une table Rendez-vous.

Ensuite je cree les champs. Chaque champ a son type, son format, ses options. Les champs texte, les champs date, les champs single select avec toutes les valeurs, les champs multi-select, les champs formule, les champs lookup qui vont chercher des infos dans une autre table, les champs rollup qui font des calculs sur les enregistrements lies. C'est la que la base prend forme. Un champ mal type ou une formule mal ecrite, ca casse tout le reste. Donc je fais ca proprement.

Je cree aussi les relations entre les tables. C'est ce qui fait qu'un contact est rattache a une entreprise, qu'un deal est rattache a un contact, qu'une relance est rattachee a un prospect. Sans ces liens, c'est juste des tableurs separes. Avec ces liens, c'est un vrai systeme ou tout est connecte.

Jour 2

Les vues et les interfaces

Le deuxieme jour, c'est la partie visible. C'est ce que le client va utiliser tous les jours.

Je cree les vues. Chaque table peut avoir plusieurs vues. Une vue kanban pour le pipeline commercial ou chaque colonne est un statut. Une vue calendrier pour voir les contenus planifies sur un mois. Une vue liste filtree pour voir uniquement les relances du jour. Une vue galerie pour voir les fiches avec des photos. Chaque vue a ses filtres, ses tris, ses regroupements, ses champs visibles. Le client ne voit que ce dont il a besoin, pas tout le reste.

Ensuite je construis les interfaces Airtable. C'est la couche par-dessus les vues qui rend tout agreable. Au lieu de naviguer dans un tableur, le client a un vrai tableau de bord. Il voit ses chiffres cles en haut, son pipeline au milieu, ses taches en cours sur le cote. Il peut cliquer sur un element pour ouvrir la fiche, modifier un champ directement depuis l'interface, passer un statut d'un clic. C'est propre, c'est clair, le client comprend tout de suite ou il est et ce qu'il doit faire.

A la fin du jour 2, la base est prete. Le client peut deja l'ouvrir, naviguer, comprendre la logique. Il manque juste les automatisations pour que ca tourne tout seul.

Jour 3

Automations natives Airtable

C'est la que la base prend vie. Sans automatisations, le client doit tout faire a la main. Avec, les choses se font toutes seules.

Le troisieme jour, je commence par les automations qui vivent directement dans Airtable. C'est les plus simples a mettre en place mais les plus utilisees au quotidien.

Par exemple : quand un prospect passe au statut "Rendez-vous pris", la date du changement se remplit automatiquement, le setter recoit une notification, et le responsable commercial voit la fiche remonter dans sa vue prioritaire.

Autre exemple : quand un contenu passe au statut "Publie", la date de publication reelle se remplit avec la date du jour, le responsable recoit un mail de confirmation, et la fiche sort automatiquement de la vue "a publier" pour aller dans la vue "publie".

Ce sont des petites actions qui, mises bout a bout, font gagner des dizaines de minutes par jour au client. Et surtout, ca evite les oublis. Un humain oublie de cocher une case, de mettre a jour un champ, de prevenir un collegue. L'automation, elle, ne loupe jamais.

Jour 4

Scenarios Make ou n8n

Le quatrieme jour, je passe aux automatisations qui connectent Airtable au reste du monde. C'est la que Make et n8n entrent en jeu.

Make et n8n, c'est des outils qui permettent de creer des ponts entre differents logiciels. Tu veux que quand un nouveau prospect arrive dans Airtable, un message soit envoye automatiquement sur le Slack du client ? C'est un scenario Make. Tu veux que quand un deal est gagne, une facture soit creee automatiquement dans Stripe ou Pennylane ? C'est un scenario Make. Tu veux que les leads qui arrivent par formulaire Typeform ou Calendly atterrissent directement dans la bonne table Airtable avec les bons champs remplis ? C'est un scenario Make.

Chaque client a ses propres outils. Certains utilisent Gmail, d'autres Outlook. Certains ont Slack, d'autres Teams. Certains sont sur HubSpot, d'autres sur Lemlist, d'autres sur LaGrowthMachine. Mon boulot, c'est de prendre les outils du client tels qu'ils sont et de les brancher proprement avec la base Airtable. Le client ne change pas ses habitudes. Ses outils communiquent entre eux, c'est tout.

Jour 5

Tests et mise en route

Le cinquieme jour, c'est le jour ou je fais tourner tout le systeme de bout en bout. Je simule des cas reels. Je cree un faux prospect, je le fais avancer dans le pipeline, je verifie que les automations se declenchent, que les mails partent, que les statuts se mettent a jour, que les notifications arrivent, que les fichiers se creent dans le bon dossier.

Si quelque chose ne marche pas comme prevu, je corrige. Si un scenario plante dans un cas particulier, je rajoute une condition. Si un mail arrive avec un mauvais format, j'ajuste le template. A la fin de la journee, tout est teste, tout tourne, et je peux te livrer le projet en disant "c'est pret, ca marche".

Les outils que j'utilise

Airtable Base de tout

C'est la ou les donnees vivent. Airtable, c'est un outil qui ressemble a un tableur mais qui fonctionne comme une vraie base de donnees. Le gros avantage par rapport a un Google Sheet, c'est les relations entre les tables, les vues multiples, les interfaces, et les automations natives. C'est l'outil parfait pour les clients qui veulent quelque chose de puissant sans avoir besoin de savoir coder.

Make ex-Integromat

C'est l'outil que j'utilise le plus souvent pour les automatisations entre Airtable et les outils tiers. Make a des centaines de connecteurs : Gmail, Slack, HubSpot, Calendly, Stripe, Notion, Google Drive, Mailchimp, Lemlist, et plein d'autres. L'avantage de Make, c'est que c'est visuel. Tu vois les scenarios sous forme de blocs relies entre eux. C'est facile a comprendre, facile a maintenir, facile a debugger si un truc plante.

n8n Open source

C'est une alternative a Make qui est open source et auto-hebergeable. Je l'utilise quand le client a des contraintes de donnees sensibles (RGPD, donnees de sante, donnees financieres) ou quand le volume d'automatisations est tellement eleve que Make deviendrait trop cher. n8n tourne sur un serveur a 10-15 euros par mois, donc les couts sont fixes quel que soit le volume.

Airtable Automations Natif

Pour les automatisations simples qui restent dans Airtable, j'utilise les automations natives. C'est plus rapide a mettre en place, ca ne depend d'aucun outil externe, et ca fonctionne directement dans la base. Pour les triggers de type "quand un champ change, fais telle action", c'est le plus efficace.

API / Webhooks Sur mesure

Quand un outil n'a pas de connecteur Make ou n8n, je passe par l'API directement. La plupart des outils SaaS modernes ont une API. Ca me permet de connecter pratiquement n'importe quoi avec Airtable, meme des outils de niche que le client utilise.
